# AI 知识助手 — 项目策划书

> 本文档为 Agent 开发指导文件，描述项目目标、功能边界、技术栈与实现要求，请严格按照本文档执行。

---

## 一、项目概述

### 1.1 项目定位

构建一个**企业级 AI 知识问答系统**，支持用户创建私有知识库、上传文档、进行多轮对话问答，并提供语音交互能力。系统后端具备完整的工程化能力（用户鉴权、限流、缓存、异步任务、评测），可作为真实 AI 产品的缩小版原型。

### 1.2 一句话描述

> 用户可以上传自己的文档，创建私有知识库，然后用自然语言（文字或语音）向知识库提问，系统基于 RAG 技术检索相关内容并生成带引用来源的回答。

### 1.3 目标用户

个人开发者、学生、小团队——需要在私有文档上做问答的场景。

## 二、技术栈

| 层级 | 选型 | 备注 |
|---|---|---|
| 后端框架 | Python + FastAPI | 异步支持，AI 生态友好 |

| 向量数据库 | Chroma | 本地持久化，按知识库隔离 |

| 部署 | Docker Compose | 所有服务容器化，一键启动 |
| 鉴权 | JWT + API Key | 双模式鉴权 |

## 三、系统功能模块

### 3.1 用户系统

**功能要求：**

- 用户注册（邮箱 + 密码，密码 bcrypt 加密存储）
- 用户登录，返回 JWT Token（有效期 7 天）
- 用户可在控制台生成、查看、撤销 API Key（格式：`sk-xxx`，用于接口调用鉴权）
- 所有 API 支持两种鉴权方式：`Authorization: Bearer <JWT>` 或 `X-API-Key: <key>`
- 接口限流：同一用户每分钟最多 30 次请求（基于 Redis 实现滑动窗口限流）

**数据库表：**

```
users: id, email, hashed_password, created_at
api_keys: id, user_id, key_hash, name, created_at, revoked_at
```

---

### 3.2 知识库管理

**功能要求：**

- 每个用户可创建多个知识库，知识库之间数据完全隔离
- 知识库支持：创建、重命名、删除
- 删除知识库时，同步删除其下所有文档及对应向量数据

**数据库表：**

```
knowledge_bases: id, user_id, name, description, created_at
```

**Chroma 隔离方式：**

每个知识库对应一个独立的 Chroma Collection，命名规则为 `kb_{knowledge_base_id}`。

---

### 3.3 文档处理（RAG Pipeline）

**支持上传格式：**

- PDF（`.pdf`）
- Markdown（`.md`）
- 纯文本（`.txt`）

**处理流程（异步执行）：**

```
上传文件
  → 文件存储（本地 /data/files 目录，Docker Volume 挂载）
  → 文本提取（PDF 用 pypdf，MD/TXT 直接读取）
  → 文本分块（chunk_size=500，overlap=50，按段落优先切分）
  → 调用阿里百炼 Embedding API 生成向量
  → 写入 Chroma（附带元数据：文件名、chunk 序号、原始文本）
  → 更新文档状态为"已就绪"
```

**文档状态机：**

`pending` → `processing` → `ready` / `failed`

**异步方案：**

上传接口立即返回 `document_id`，实际处理任务通过 Redis 队列（`rq` 或 `celery`）异步执行，前端可轮询状态接口。

**数据库表：**

```
documents: id, kb_id, user_id, filename, file_path, status, chunk_count, created_at, error_msg
```

---

### 3.4 对话与问答

**功能要求：**

- 用户在指定知识库下创建会话（Session）
- 每次提问触发完整 RAG 流程
- 回答必须附带引用来源（文件名 + 原始片段）
- 支持多轮对话上下文
- 当会话历史超过 10 轮时，自动对历史消息进行摘要压缩，保留最近 3 轮原文 + 摘要文本，避免超出 token 限制

**RAG 检索流程：**

```
用户提问
  → 检索最近对话历史（最多 10 轮）
  → 若超过 10 轮，触发记忆压缩（调用 LLM 生成摘要）
  → 将问题向量化（Embedding API）
  → 在对应 Chroma Collection 中检索 Top-5 相似片段
  → 构建 Prompt（系统提示 + 历史摘要 + 检索片段 + 当前问题）
  → 调用 LLM 生成回答（流式输出）
  → 将引用片段与回答一并返回
  → 存储本轮问答到数据库
```

**Prompt 模板（系统提示）：**

```
你是一个知识库问答助手。请根据以下参考资料回答用户的问题。
回答时必须在末尾列出引用来源（格式：[来源：文件名，第X段]）。
如果参考资料中没有相关内容，请明确告知用户"知识库中暂无相关信息"，不要编造答案。

参考资料：
{context}
```

**数据库表：**

```
sessions: id, user_id, kb_id, title, summary, created_at, updated_at
messages: id, session_id, role(user/assistant), content, sources(JSON), created_at
```

---

### 3.5 语音交互

**STT（语音转文字）：**

- 接收前端上传的音频文件（支持 wav / mp3 / m4a）
- 调用阿里百炼语音识别 API 返回文字
- 返回识别文本，由前端填入对话输入框（或直接触发问答）

**TTS（文字转语音）：**

- 接收回答文本
- 调用阿里百炼语音合成 API 返回音频流
- 以流式方式返回 mp3 音频数据

**接口设计：**

```
POST /api/voice/stt   上传音频 → 返回文字
POST /api/voice/tts   传入文字 → 返回音频流
```

---

### 3.6 评测系统

**功能要求：**

- 每条 assistant 回答支持用户点赞（👍）或踩（👎）
- 记录每次 RAG 检索命中的来源文档片段，可回溯"这个回答从哪段内容得出"
- 后台统计指标（管理员或用户本人可查）：
  - 好评率（点赞数 / 总评价数）
  - 平均响应时间（从提问到完整回答的耗时）
  - 各知识库的问答总量

**数据库表：**

```
feedbacks: id, message_id, user_id, rating(1/-1), created_at
retrieval_logs: id, message_id, document_id, chunk_index, chunk_text, score, created_at
```

## 四、API 接口总览

### 用户与鉴权

```
POST   /api/auth/register          注册
POST   /api/auth/login             登录，返回 JWT
GET    /api/auth/me                获取当前用户信息
POST   /api/auth/api-keys          创建 API Key
GET    /api/auth/api-keys          列出所有 API Key
DELETE /api/auth/api-keys/{id}     撤销 API Key
```

### 知识库

```
POST   /api/kb                     创建知识库
GET    /api/kb                     列出当前用户的知识库
PATCH  /api/kb/{kb_id}             重命名知识库
DELETE /api/kb/{kb_id}             删除知识库
```

### 文档

```
POST   /api/kb/{kb_id}/documents          上传文档（异步处理）
GET    /api/kb/{kb_id}/documents          列出知识库下所有文档
GET    /api/kb/{kb_id}/documents/{doc_id} 查询文档处理状态
DELETE /api/kb/{kb_id}/documents/{doc_id} 删除文档
```

### 对话

```
POST   /api/kb/{kb_id}/sessions           创建会话
GET    /api/kb/{kb_id}/sessions           列出会话
GET    /api/sessions/{session_id}/messages 获取历史消息
POST   /api/sessions/{session_id}/chat    发送消息（流式返回）
DELETE /api/sessions/{session_id}         删除会话
```

### 语音

```
POST   /api/voice/stt              语音转文字
POST   /api/voice/tts              文字转语音（流式返回）
```

### 评测

```
POST   /api/messages/{message_id}/feedback   提交点赞/踩
GET    /api/kb/{kb_id}/stats                 获取知识库统计数据
GET    /api/messages/{message_id}/sources    查看检索来源
```

## 六、Docker Compose 服务构成

```yaml
services:
  backend:      # FastAPI 应用，端口 8000
  worker:       # 异步任务 worker（同一代码库，不同启动命令）
  postgres:     # PostgreSQL 15
  redis:        # Redis 7
  chroma:       # Chroma 向量数据库（官方 Docker 镜像）
```

所有数据目录通过 Volume 挂载到宿主机 `./data`，保证数据持久化。

## 七、环境变量说明（.env）

```
# 数据库
DATABASE_URL=postgresql://user:pass@postgres:5432/ai_assistant

# Redis
REDIS_URL=redis://redis:6379/0

# LLM（OpenAI 兼容接口）
LLM_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
LLM_API_KEY=your_dashscope_api_key

# Embedding
EMBEDDING_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
EMBEDDING_API_KEY=your_dashscope_api_key
EMBEDDING_MODEL=text-embedding-v3

# 阿里百炼语音
DASHSCOPE_API_KEY=your_dashscope_api_key
STT_MODEL=paraformer-realtime-v2
TTS_MODEL=cosyvoice-v1

# JWT
JWT_SECRET=your_random_secret_string
JWT_EXPIRE_DAYS=7

# 限流
RATE_LIMIT_PER_MINUTE=30
```

## 八、非功能性要求

| 项目 | 要求 |
|---|---|
| 错误处理 | 所有接口返回统一错误格式 `{code, message, detail}` |

## 十、验收标准

完成后应满足以下所有条件：

- [ ] `docker-compose up` 一键启动所有服务，无报错
- [ ] 可通过注册/登录获取 JWT，可生成 API Key
- [ ] 创建知识库后，上传 PDF / MD / TXT，文档状态最终变为 `ready`
- [ ] 在已有文档的知识库中提问，回答包含引用来源
- [ ] 多轮对话超过 10 轮后，系统自动压缩历史，接口仍正常响应
- [ ] 语音接口可正常调用（STT 返回文字，TTS 返回音频流）
- [ ] 对回答点赞/踩后，统计接口数据变化正确
- [ ] 限流生效：同一用户 1 分钟内超过 30 次请求返回 429
- [ ] 所有接口有 Swagger 文档（FastAPI 自动生成，访问 `/docs`）

### 11.2 自动提交规范

每次**重大变更**完成后，必须按以下流程执行：

1. **更新 README.md**，反映当前项目最新状态（已实现功能、启动方式、环境变量说明等）
2. **提交变更**，commit message 使用约定式提交格式（Conventional Commits）：

```
feat: 添加 RAG 检索核心流程
fix: 修复文档处理状态未更新的问题
docs: 更新 README 部署说明
refactor: 重构 embedding_service 调用方式
chore: 添加 docker-compose 健康检查
```

3. **推送到远程**：

```fish
git add .
git commit -m "feat: xxx"
git push origin main
```

### 11.3 重大变更的定义

以下情况视为重大变更，必须触发提交：

- 完成一个新功能模块（如用户系统、文档上传、RAG 问答等）
- 修改了 `docker-compose.yml` 或 `Dockerfile`
- 修改了 `.env.example`
- 完成一个阶段性功能并通过本地验证
